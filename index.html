설계 자동화가 아닌, 구조화 및 설명 가능한 프로세스로의 전환

2026년 3월 24일
1

추진 배경 및 문제 인식

현재 UX 설계는 표면적으로 논리적인 과정으로 보이나, 실제 수행 과정에서는 다음과 같은 구조적 한계가 존재합니다.

  • 기준 부재: 초기 단계에서 개인의 경험과 감에 의존하는 비중이 높음
  • 일관성 부족: 유사 프로젝트임에도 불구하고 설계 방식과 결과의 편차 발생
  • 사후 정당화: 설계 의사결정 과정에서 근거가 결과 도출 후 사후적으로 정리됨

특히 공공 프로젝트의 리스크:
다양한 이해관계자와 설명 책임 요구로 인해, 설계 과정의 '근거 부족'이 곧 프로젝트 리스크로 직결되는 구조입니다.
2

UX 설계는 왜 '근거'가 필수적인가

UX 설계는 단순한 화면 구성이 아닌, 사용자 행동을 설계하고 서비스 성과를 결정하는 의사결정 과정입니다.

선택의 연속

정보의 순서, 기능 배치, 사용자 흐름 등 모든 설계는 선택입니다. 근거 없는 선택은 개인의 취향에 불과합니다.

설명 가능성 (Explainability)

공공 프로젝트에서 발주처와 이해관계자에게 "왜 이렇게 설계했는가"를 논리적으로 설명할 수 있어야 합니다.

일관성 및 재현성

설계 기준이 개인에게 종속되지 않아야, 동일한 문제에 대해 일관된 품질의 구조를 도출할 수 있습니다.

공공의 책임

공공 UX는 단순 편의를 넘어 정책 전달과 형평성을 다룹니다. "잘 만든 것"보다 "정당화될 수 있는 것"이어야 합니다.

제안 방향: 근거 기반 구조 설계

본 기획안은 UX 설계의 완전 자동화를 목표로 하지 않습니다.
대신, AI를 활용하여 설계의 '근거와 구조'를 먼저 생성하고, 기획자는 이를 기반으로 설계를 완성하는 프로세스를 제안합니다.

핵심 변화 포인트

Before (기존 방식)

설계 후 → 근거 정리

  • 결과물 중심 접근
  • 사후 논리화 (Reverse Engineering)
  • 개인의 역량에 의존
After (제안 방식)

근거 생성 → 구조 설계

  • 구조(Structure) 중심 접근
  • 사전 논리화 및 기준 수립
  • 시스템 기반의 일관성
7

핵심 구성 요소

7.1 AI Agent (근거 생성 엔진)

요구사항을 분석하여 UX 목표와 UI 구조(Layout)를 텍스트 및 코드 형태로 산출합니다.

[Output Example] Screen: Home 목표: 공공 서비스 상품 탐색 효율 향상 및 정책 인지 강화 구조 (Structure Logic): 1. Global Header: 브랜드 신뢰도 확보 및 전체 메뉴 접근성 2. Unified Search Bar: 목적형 사용자의 빠른 진입 유도 3. Category Tabs (Sticky): 5개 주요 민원 분야로 탐색 경로 분기 4. Card List (2 Col): 핵심 서비스 직관적 노출 (이미지+타이틀) 5. Floating CTA: 긴급 민원 신청 접근성 보장

7.2 Figma 설계 시스템 (구현 체계)

AI가 제안한 구조를 즉시 시각화할 수 있도록 블록(Block) 단위의 UI 컴포넌트 시스템을 구축하여 반복 조립 가능한 환경을 제공합니다.

8

왜 '완전 자동화'가 아닌가

AI는 강력한 도구이지만, 최종 책임과 판단은 기획자에게 있습니다.

맥락적 의사결정

서비스의 숨겨진 맥락과 비정형적 제약조건은 AI가 완전히 파악하기 어렵습니다.

공공의 특수성

정책적 민감도와 행정적 제약사항은 인간의 정무적 판단이 필요합니다.

설득과 책임

결과물에 대해 이해관계자를 설득하고 책임지는 주체는 결국 사람입니다.
